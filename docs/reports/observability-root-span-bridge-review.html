1. 变更概述

本轮变更解决了 Phoenix Sessions 中 AI 回复显示 undefined 的问题。

根因：router.deliver_to_agent root span 通过 withSpan() 自动管理生命周期，在 wakeContainer() 返回后立即 span.end()。Delivery 是异步的（1s+ 后 poll 触发），此时 root span 已关闭，output.value 无法写入。Phoenix Sessions 只读 root span 的 output.value。

修复方案：将 root span 生命周期从 router 延伸到 delivery 完成。

4. 潜在风险与缓解

4.1 Span 泄漏（Memory Leak）

风险：如果 delivery 永远不 poll 到某个 session（例如 session 被归档但 span 未 end），rootSpanBridge Map 会持有 Span 引用。

缓解：

  • container.on('close') 和 container.on('error') 已覆盖所有容器退出路径
  • 正常 idle-exit（code=0）不调用 fail（因为 delivery 会先完成）
  • 如果需要更强保证，可在 host-sweep 中加 TTL 清理（当前未实现，风险极低）

4.2 Span 双重 End

风险：OTel SDK 对已 end 的 span 再次 end 的行为是 undefined。

缓解：endSessionRootSpan 和 failSessionRootSpan 都先从 Map 中 delete 再 end，第二次调用时 Map 中已无条目，直接 return。

4.3 Output 截断

风险：lastDeliveredText 只取最后一条用户可见消息。多条回复时只保留最后一条。

缓解：Phoenix Sessions 的 AI 列只显示一条文本，取最后一条是合理行为。如需完整对话，应查看 trace 内的各 delivery.channel.send 子 span。

7. 审批建议

  • 代码逻辑正确，时序安全
  • 所有退出路径都有 span 兜底
  • 无 AI slop（无多余注释、无冗余抽象、无过度工程）
  • 与现有 observability schema 兼容
  • 建议后续：host-sweep 中加 rootSpanBridge TTL 清理（P2，非阻塞）
