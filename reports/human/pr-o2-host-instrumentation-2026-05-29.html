PR-O2 Host Instrumentation Report

Date: May 29, 2026

Executive Summary

This report outlines the successful implementation of host-side OpenTelemetry instrumentation for PR-O2. The core observability framework was established, and distributed tracing was wired across the host routing, delivery, and container management subsystems. Critical QA bugs involving trace context loss during asynchronous delivery loops and exporter misconfigurations were identified and fully resolved.

What Changed

The following structural updates were applied to the host to enable OpenTelemetry:

  • Implemented core observability utilities for trace initialization and context propagation.
  • Instrumented routing (router.route, router.deliver_to_agent).
  • Instrumented delivery pipelines (delivery.session.drain, delivery.message.deliver).
  • Instrumented incoming channels (cli.event.received) and outgoing responses (delivery.channel.send).
  • Instrumented container management boundaries (container.spawn, container.wake).
  • Patched trace configuration to correctly export to the Arize Phoenix backend.

Root Cause Found During QA

During testing, multiple issues prevented reliable trace propagation and exporting. The root causes and their respective fixes are documented below.

1. Session Context Loss

Cause: consumeSessionSpanContext() deleted the bridged session context during empty delivery polls, which happened before the agent actually wrote outbound messages.

Fix: Introduced a new getSessionSpanContext() peek API. Context is now cleared strictly after the actual outbound work is handled.

2. Exporter Failure

Cause: The @opentelemetry/exporter-trace-otlp-http module produced a 415 Unsupported Media Type error when reporting to Phoenix.

3. Sampler Trace Drop

Cause: The custom sampler returned decision: 1 (which corresponds to RECORD), instead of RECORD_AND_SAMPLED. This resulted in spans being recorded locally but not exported to the collector.
