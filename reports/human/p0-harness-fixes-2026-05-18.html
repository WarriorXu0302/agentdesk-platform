§1 背景回顾

之前执行的 MUAP Dev-Harness 审查（依据 reports/muap-harness-scorecard.html）暴露了严重的上下文连续性问题。评估得出总分 21/35（60%），但最致命的是“决策留痕”项仅得 1★，且跨仓迁移宪法的发现性为“∞ 跳”。具体表现为：当外部 coding agent 首次进入本仓库时，无论通过 README.md、CLAUDE.md 还是 AGENTS.md 这三条标准入场路径，都无法在 3 跳内发现 V1→MUAP 的迁移宪法（v1.2）；同时，ADR（架构决策记录）档案完全空白，只有直接对 docs/ 目录进行主动探索时才能偶然命中关键指引。这会导致新 agent 反复陷入重新发明轮子或破坏已有契约的困境。

  • Decision logging is non-optional: 明确 ADR 不可或缺，任何涉及选择或移除的设计决议都必须记录。

5.3 宪法级 ADR 卡片

ADR-0001: 整体迁移路径（Q1 镜像） Accepted

Context 摘要：V1 承载了大量契约与组件，MUAP 需要继承产物但不背运行期包袱。

Options 简表：A) 渐进拆解+重新组装；B) V1整体复制翻新；C) 完全重写。

Decision：采用 Option A。V1 prompt 落入 CLAUDE.local.md，Python 技能走 ERP gateway，V1 内部状态(SOUL/USER)不进 MUAP。

Key Consequences：保持 MUAP 企业基线纯净，但需要长期跨仓维护边界。

ADR-0002: 立即启动迁移（Q2 镜像） Accepted

Context 摘要：确认是等 V1 完全闭环还是立即启动 Phase 0a 迁移。

Options 简表：A) 立即启动并并行 Phase 0b/0.5；B) 等 V1 收口；C) 仅启动 0a。

Decision：采用 Option A。避免 V1 状态漂移成本，利用团队最高记忆度。

Key Consequences：需要隔离 V1 与 MUAP 开发状态，承受短期并行复杂度，换取项目周期可控。
