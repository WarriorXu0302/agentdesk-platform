MUAP 架构解释 — 三个核心概念

双 Frontdesk · Lab Desk · ERP Gateway — Phase 0a 引入的企业级多 Agent 拓扑
📅 2026-05-20 📦 Phase 0a (commit dba7249) 📋 ADR-0008 / ADR-0009
一句话总结 · MUAP 不是一个 Agent，而是一群 Agent 的调度平台。 "Frontdesk" 是接待用户的入口 Agent；"双 Frontdesk" 是两种不同风格的入口并存； "Lab Desk" 是第二种入口（实验室专用）；"ERP Gateway" 是所有 Agent 共享的后端服务接口。

§1 双 Frontdesk — 两种接待风格并存

什么是 Frontdesk？

Frontdesk 是用户（飞书/CLI）进入 MUAP 系统的第一个 Agent。 它负责接待用户、理解意图，然后决定：自己处理，还是派给专门的 Worker Agent。

为什么要"双"？

企业场景下，不同部门的交互模式完全不同。 Phase 0a 引入了两种 Frontdesk 并存的设计，让不同团队可以用最适合自己的方式使用 Agent 平台。

维度 Primary frontlane-template-frontdesk Secondary frontlane-lab-frontdesk
角色定位 企业客服代表 — 接待、分类、派活 实验室助手 — 自包含，直接执行
Worker 关系 拥有 5 个共享 Worker（access / sales / finance / approval / ops） 无 Worker — 一切自己做
处理模式 委托模型：分类意图 → 路由到专业 Worker 直驱模型：直接调用 ERP Gateway 完成操作
语言 / 提示词 英文 · 通用企业 ERP 提示词 中文 · V1 "小环"实验室助手提示词（含完整路由表）
资源配额 768 MB RAM / 1 CPU / 384 PIDs 1024 MB RAM / 1 CPU / 512 PIDs（更高，因直驱复杂度）
通道入口 默认接收飞书/CLI 消息（autowire 自动连接） 不自动接收 — 需显式设置 ENTERPRISE_FRONTDESK_FOLDER=frontlane-lab-frontdesk
Git 追踪 不追踪（运行时生成） 版本控制（prompt 契约可审计）
适用场景 通用企业入口 — 客服、审批、销售、财务 实验室专用 — 科研情报、设备操控、实验监控

路由器知道"双 Frontdesk"吗？

不知道 · src/router.ts 和 src/delivery.ts 对所有 Agent 一视同仁。 "谁接收哪些消息" 完全由布线层（messaging_group_agents 数据库表）决定， 而非路由器代码。这保证了添加更多 Frontdesk 时不需要改核心路由逻辑。

§2 Lab Desk — 实验室专用的"小环"

Lab Desk 是什么？

Lab Desk（frontlane-lab-frontdesk）是第二种 Frontdesk 的具体实例。 它继承自 V1 的"小环"实验室助手，专门为物理实验室场景设计。

Lab Desk 的能力

  • 科研情报：文献检索、技术咨询、数据分析
  • 物理设备操控：移液器、机械臂、云台等实验室设备（通过 MCP 工具）
  • 远程桌面：通过 noVNC 接管实验电脑
  • 实验监控：轮询摄像头、检测异常
  • 飞书办公：日程、任务、文档、考勤

Lab Desk 的 container.json 长什么样？

{
  "enterpriseGateway": {
    "baseUrl": "http://localhost:8088",
    "timeoutMs": 30000,
    "defaultHeaders": {
      "X-FrontLane-Source": "frontlane-lab-frontdesk"  // ← 身份标记
    }
  },
  "memoryMode": "erp",
  "a2aSessionMode": "root-session",
  "resources": { "memoryMb": 1024, "cpus": 1, "pidsLimit": 512 }
}
关键设计 · X-FrontLane-Source: frontlane-lab-frontdesk 这个 header 是 Lab Desk 的身份标记。 ERP Gateway 收到请求时，可以通过这个 header 区分"这个请求来自哪个 Frontdesk"，从而应用不同的权限策略。 Phase 0a 的一个关键修复（header merge bug）就是为了保证这个标记不被覆盖。

为什么 Lab Desk 不接 Worker？

  • 命名冲突：Worker 会创建一个叫 frontdesk 的反向路由别名。如果两个 Frontdesk 都有 Worker，这个别名会冲突
  • 语义对齐：V1 的"小环"就是直驱 ERP 的，不需要中间 Worker
  • 职责清晰：Lab Desk 专注实验室场景，不混入通用客服逻辑

§3 ERP Gateway — Agent 与企业后端的桥梁

为什么需要 Gateway？

Agent（LLM）本身不懂你的 ERP 系统。 如果让每个 Agent 直接对接不同 ERP 的私有 API，会导致：

  • Agent 提示词要写死特定 ERP 的接口细节
  • 换一个 ERP 系统就要重写所有 Agent
  • 权限、审计、幂等逻辑散落在各个 Agent 里

Gateway 的解决方案：在 Agent 和 ERP 之间加一层统一的 HTTP 接口。 Agent 只需要调用 5 个标准工具，Gateway 负责翻译成具体 ERP 的 API。

Gateway 暴露哪些接口？

端点用途谁调用
POST /describe 查询 Gateway 支持哪些操作、需要哪些字段 Agent（了解能力边界）
POST /authorize 检查当前用户是否有权限执行某操作 Agent（执行前校验）
POST /execute 执行业务操作（创建订单、审批付款等） Agent（核心业务调用）
POST /memory/get 读取用户的长期记忆（偏好、历史、摘要） Agent（个性化服务）
POST /memory/upsert 写入/更新用户的长期记忆 Agent（学习用户偏好）

Gateway 的安全模型

1. 身份传递（requesterSource）

用户发消息
→
Host 提取身份
→
写入 requesterSource: "session"
→
Gateway 信任

如果身份无法确定（定时任务、无归属的 a2a 调用），requesterSource 会是 "agent-asserted"， Gateway 应该拒绝写操作。

2. HMAC 签名（可选）

配置 signingKey 后，每个 Gateway 请求会携带三个 header：

  • x-frontlane-timestamp — unix 时间戳
  • x-frontlane-nonce — 32 字符随机串
  • x-frontlane-signature — HMAC-SHA256 签名

Gateway 可以验证签名，拒绝伪造或重放的请求。

3. 审计追踪（erp_audit）

每次 Gateway 调用（无论成功失败）都会在 Host 的 erp_audit 表写一行记录： 谁、什么时候、调了什么、结果如何、耗时多久。

Gateway 的数据流

┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│  飞书用户    │     │  Frontdesk  │     │   Worker    │
│  (员工 A)   │────▶│  (接待)    │────▶│  (执行)    │

职责分离 · Agent 平台负责"聊天、推理、派活"；ERP 后端负责"权限、审批、审计、业务逻辑"。 两者通过 Gateway 的 5 个标准端点解耦，换 ERP 只需要换 Gateway 实现，Agent 层完全不变。

configure-enterprise-gateway.ts 做了什么？

这个脚本把 Gateway 的 URL、超时、自定义 header 写入每个 Agent Group 的 container.json。 默认会配置所有企业组（2 个 Frontdesk + 5 个 Worker）。 

# 配置所有企业组连到同一个 Gateway
pnpm configure:enterprise-gateway --base-url https://erp-gateway.internal/api/agent

# 只配特定组
pnpm configure:enterprise-gateway \
  --base-url https://erp-gateway.internal/api/agent \
  --folders frontlane-template-frontdesk,frontlane-lab-frontdesk \
  --timeout-ms 20000 \
  --header x-tenant=erp-a
Phase 0a 关键修复 · 这个脚本原来用 replace 语义（整个 enterpriseGateway 对象被覆盖）， 会丢失 Lab Desk 的 X-FrontLane-Source 标记。Phase 0a 修复为 merge 语义（保留已有的 header，只覆盖 CLI 指定的）。

§5 关键设计决策（ADR 摘要）

决策选择理由
默认拓扑 双 Frontdesk（Primary + Lab） V1/V2 灰度过渡期需要并存；一条命令拉起全部
Lab Desk 是否接 Worker 不接 避免 frontdesk 反向路由别名冲突；语义对齐 V1
Lab Desk 入 git 方式 .gitignore 精确例外 Canonical 文件（prompt + config）应该版本控制、可审计
Gateway header 行为 Merge（保留已有 + 覆盖 CLI 指定） 防止 X-FrontLane-Source 被 configure-enterprise-gateway 覆盖
env 是否自动写 只写 .env.local.proposed + diff 报告 保留人工 redaction policy；<PLACEHOLDER> 不许被 agent 替换
Gateway 端点 5 个标准端点（describe / authorize / execute / memory/*） 所有 ERP 共用同一接口；换 ERP 只换 Gateway 实现

§6 常见问题

Q: 我可以只用一个 Frontdesk 吗？

可以。用 --frontdesk-folder frontlane-template-frontdesk 参数运行 pnpm init:enterprise， 就只创建 Primary Desk，不创建 Lab Desk。这是 V1 的向后兼容模式。

Q: 怎么让 Lab Desk 接收飞书消息？

设置环境变量 ENTERPRISE_FRONTDESK_FOLDER=frontlane-lab-frontdesk， 然后重启。Autowire 机制会把飞书通道连接到 Lab Desk。

Q: Gateway 必须自己实现吗？

是的。Gateway 是你自己的后端服务，负责对接你的 ERP 系统。 MUAP 只定义了 Gateway 的接口契约（5 个端点 + 请求/响应格式），不提供实现。

Q: 可以没有 Gateway 吗？

可以。把 container.json 里的 enterpriseGateway 删掉， Agent 就不会调用任何 ERP 操作。但这意味着 Agent 只能聊天，不能执行业务动作。

Q: Agent 怎么知道用户是谁？

Host 在每批请求开始时，从飞书消息中提取用户身份，写入 requesterSource: "session" + requester.userId。 Agent 无法伪造这个身份——它是由容器运行时在 batch 开始时固定的。 即使 Frontdesk 派活给 Worker，原始用户身份也会通过 origin_user_id 传递到 3 跳深的 Worker。
