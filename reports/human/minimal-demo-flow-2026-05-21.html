§1 三条候选路径 · 选择矩阵

"最小完整"有三种合理解读，按"附加依赖"递增排序如下。本报告把 Path A 作为推荐演示路径。

路径 能演示什么 依赖 典型耗时 适用场景
Path A
推荐		 消息 → 路由 → 容器 → 模拟回复 → 回投。验证 host/runner 协议是否正常。 Node 20 · Docker ≈ 5 分钟 新人 onboarding · CI smoke · "代码能跑起来吗"自检
Path B Path A + 真实 Claude / OpenAI 推理（让 Agent 自然回复一句话） Path A + 一个 Anthropic 或 OpenAI 兼容 API key ≈ 7 分钟 给非工程师演示 "AI 能聊天" · UI 评审
Path C Path B + 一次 ERP gateway 调用（erp_describe） Path B + 一个 echo HTTP server (any) · 可手撸 30 行 Node ≈ 12 分钟 给企业方演示 "Agent ↔ ERP" 信任链
不在演示范围内 · 飞书入口（需 cli_xxx app id + 公网回调）/ Phoenix-Grafana observability（PR-O1 已交付但与单 agent 演示正交）/ 多用户隔离对照（需要至少两位真实身份）/ HMAC 签名 / `erp_audit` 审计 · 这五项是企业部署关注，但不是"看 demo 能跑"必需的。

§2 Path A · 推荐演示流程（CLI + mock provider · 离线）

下面 5 步严格按顺序执行，期望输出已逐条标注。任意一步失败请停下来排查再往下走。

Step 1

克隆 + 安装依赖

≈ 60 s
仓库根目录下安装 pnpm 工作区依赖。pnpm install 同时会处理 better-sqlite3 的 native rebuild — 如果 Node ABI 不匹配会在这一步报错。
pnpm install
期望尾行：Done in <n>s · 后续步骤都依赖 node_modules 与 scripts/ 可执行。
Step 2

构建 agent runner 容器镜像

首次 ≈ 3-5 分钟，后续命中 layer 缓存秒级
host 启动 session 时会用 docker run --rm muap-runner:<slug> 拉起一个隔离的 agent runner 容器；该镜像包含 container/agent-runner/ 编译产物以及 mock / openai / claude 三个 provider 注册器。镜像 tag 由仓库绝对路径的 sha 前 8 位派生，所以两个 checkout 不会互踩。
pnpm container:build
期望尾行：Successfully tagged muap-runner:<8-hex-slug> · 之后 docker images | grep muap-runner 应能看到该 tag。
Step 3

初始化最小 CLI agent + 锁定 mock provider

≈ 5 s · 幂等
这一步做三件事：① 在中央 SQLite 创建一个 agent group cli-with-demo；② 创建 cli/local messaging group；③ 把这个 messaging group wire 到 agent group。脚本会在 groups/cli-with-demo/ 写出 CLAUDE.md 和 container.json。
# 1. 跑 init 脚本
pnpm exec tsx scripts/init-cli-agent.ts \
  --display-name "Demo User" \
  --agent-name "Demo" \
  --folder cli-with-demo

# 2. 把 provider 切成 mock（默认是 'claude'，会要求 ANTHROPIC_API_KEY）
# 用 jq 原地修改，没有 jq 就手工 vi groups/cli-with-demo/container.json
node -e "const f='groups/cli-with-demo/container.json'; const o=JSON.parse(require('fs').readFileSync(f,'utf8')); o.provider='mock'; require('fs').writeFileSync(f, JSON.stringify(o,null,2)+'\n');"
期望输出（部分）：
Created agent group: ag-…
Created CLI messaging group: mg-…
Wired cli: mg-… -> ag-…
验证：cat groups/cli-with-demo/container.json | grep provider 应包含 "provider": "mock"。
Step 4

启动 host service（一个独立终端）

长驻进程 · Ctrl+C 停
host 会：① 跑 SQLite migration；② 启动 channel adapter（cli + feishu，但 feishu 未配 env 时静默 noop）；③ 启动 host-sweep 60 s 周期；④ 在 data/cli.sock 监听 Unix socket。
pnpm dev
期望日志（关键 3 行）：
FrontLane Agent Platform starting
Central DB ready { path: '…/data/v2.db' }
cli channel listening at …/data/cli.sock
看到第三行就可以打开第二个终端做 Step 5。
Step 5

用 CLI client 发一句话 · 收到 mock 回复

≈ 10 s（含首次 docker 容器冷启）
在第二个终端运行 pnpm chat，它会连接 data/cli.sock 写入一个 {"text":"…"} JSON 行。host router 把这条消息塞进 session queue，container-runtime 拉起 docker，agent runner poll 到消息后调 mock provider，返回 "Mock response to: hello"，host 写回 socket。
pnpm chat hello from minimal demo
期望终端输出（client 侧）：
Mock response to: hello from minimal demo
同时第一个终端的 host 日志应能看到（关键事件）：
Inbound message routed → Container started → Provider event { type: 'result', text: 'Mock response …' } → Delivered response。
✅ 演示通过判定 · Step 5 客户端打印出 Mock response to: … · host 日志中能看到 Container started + Delivered response 两行 · 进程退出码 0。任意一项缺失 → 见 §6 排错。

§4 升级到 Path B / Path C 的最小增量

B. 接真实 LLM · 让 agent 自然说话（不是 canned 回复）

把 Path A Step 3 第二条命令换成：把 provider 设成 openai 或 claude，并提供对应的 env。

# 选项 B-1：OpenAI 兼容（任何 baseURL，比如 d1token / Azure / 本地 vLLM）
export OPENAI_BASE_URL=https://www.d1token.com/
export OPENAI_API_KEY=sk-xxx
export OPENAI_MODEL=gpt-5.4
node -e "const f='groups/cli-with-demo/container.json'; const o=JSON.parse(require('fs').readFileSync(f,'utf8')); o.provider='openai'; require('fs').writeFileSync(f, JSON.stringify(o,null,2)+'\n');"

# 选项 B-2：原生 Claude SDK
export ANTHROPIC_API_KEY=sk-ant-…
node -e "const f='groups/cli-with-demo/container.json'; const o=JSON.parse(require('fs').readFileSync(f,'utf8')); o.provider='claude'; require('fs').writeFileSync(f, JSON.stringify(o,null,2)+'\n');"

其余 Step 1/2/4/5 不变。预期 Step 5 输出从 Mock response to: … 变成真实模型的自由回复（Demo 这个 agent 会按 CLAUDE.md 自我介绍并邀请你接着聊）。

C. 接 ERP gateway · 演示企业信任链最小切片

在 groups/cli-with-demo/container.json 顶层加 enterpriseGateway.baseUrl，让容器内的 erp_* MCP 工具有可达后端。最小可演示后端 = 一个 Node echo 服务：

# 1. 起 echo 服务（终端 3）
node -e "require('http').createServer((q,s)=>{let b='';q.on('data',d=>b+=d);q.on('end',()=>{console.log(q.method,q.url,b);s.setHeader('content-type','application/json');s.end('{\"ok\":true,\"echo\":'+JSON.stringify(b)+'}')})}).listen(8088)"

# 2. 把 baseUrl 写入 container.json
node -e "const f='groups/cli-with-demo/container.json'; const o=JSON.parse(require('fs').readFileSync(f,'utf8')); o.enterpriseGateway={baseUrl:'http://host.docker.internal:8088', timeoutMs:5000, defaultHeaders:{'X-FrontLane-Source':'cli-with-demo'}}; o.memoryMode='erp'; require('fs').writeFileSync(f, JSON.stringify(o,null,2)+'\n');"

# 3. Path A Step 5 换成显式触发 erp_describe（自然语言提示）
pnpm chat "call the erp gateway describe endpoint with operation list_orders"

期望终端 3 上看到 echo 服务打印一行 POST /describe …，包含 host 注入的身份头：x-frontlane-user-id、x-frontlane-requester-source、x-frontlane-origin-user-id，以及（若开启 HMAC）x-frontlane-signature。central DB 的 erp_audit 表会落一行成功记录：

sqlite3 data/v2.db "select user_id, operation, requester_source, http_status from erp_audit order by created_at desc limit 1;"

§6 常见排错

症状原因处置
FrontLane daemon not reachable at .../cli.sock host 没在跑，或 host 在不同 DATA_DIR 确认 Step 4 仍在前台；如改了 DATA_DIR，client 也要相同 env
Step 5 一直没回复，30 s 后客户端超时 容器镜像没 build 或 build 出 ABI 不匹配 另开终端：docker images | grep muap-runner 必须有 tag；没有就重跑 Step 2
host 日志报 provider 'claude' requires ANTHROPIC_API_KEY Step 3 第二条命令没跑，provider 还是默认值 claude 检查 groups/cli-with-demo/container.json 的 provider 字段
Step 4 报 circuit breaker open: backoff 5s 短时间反复重启被启动 backoff 拦下 等几秒；clean 退出会自动 reset，强制 reset：rm data/circuit-breaker.json
Path C echo 服务收不到请求 容器到 host loopback 路径不对（Linux 上 host.docker.internal 不一定可达） Linux 用 --add-host=host.docker.internal:host-gateway 或改成 host 的 LAN IP
Path C erp_audit 一行没有 agent 没真正去调 erp_describe（可能直接用普通对话回答了） 提示词改成更指令式："please invoke the erp_describe MCP tool with {operation:'list_orders'} and report the JSON response"

§8 与"生产可用"之间的差距

本报告刻意把演示路径压到最小。要进入生产 / 准生产，至少还要：

  • 跑 pnpm setup:lab-frontdesk（前台 + 5 worker + ERP gateway header），见 Phase 0a 报告
  • 配置飞书：FEISHU_APP_ID / FEISHU_APP_SECRET / FEISHU_EVENT_MODE / FEISHU_BOT_OPEN_ID（参考 docs/feishu-channel.md）
  • 启用 observability：pnpm obs:up 起 Phoenix + Postgres + Grafana（PR-O1 交付，11/11 manual QA 已 PASS）
  • 启用 HMAC：在 ERP gateway 配 signingKey，host 与 gateway 双侧持有
  • 配置 session TTL / 容器并发上限：FRONTLANE_SESSION_TTL_DAYS / MAX_CONCURRENT_CONTAINERS / FRONTLANE_IDLE_EXIT_MS
  • 等待 PR-O2（host instrumentation）+ PR-O3（runner manual spans，Oracle ruling）落地后才有完整 trace 视图
定位 · 这条最小演示路径是 "baseline 健康自检" 的工具，不是 "产品 demo"。前者验证代码能跑；后者要展示业务价值。两者的取舍清晰且无冲突。
