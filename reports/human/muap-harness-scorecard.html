MUAP Agent Spec Harness — Maturity Scorecard

Project: MultiUserAgentPlatform (MUAP) — /Users/desjajja/Dev/realityloop/MultiUserAgentPlatform Version evaluated: v2.0.44 Date: 2026-05-18 Reference benchmarks: taac2026 (/Users/desjajja/Dev/taac2026); LangSmith / Mastra / Inspect AI / mem0 / Letta / Zep / Braintrust / Confident AI / Blaxel (2025-2026 SOTA)

Executive Summary

MUAP is a production-ready enterprise agent runtime with industry-leading session isolation and a clean dual-SQLite communication contract, but lacks the evaluation harness, prompt versioning, and CI/CD discipline found in mature 2026 agent platforms. The biggest single gap is the absence of any behavioral evaluation system — the classification-log table is explicitly designed as a regression-suite seed but has no consumer.

Dimension Details

1. Agent Specification & Prompt Versioning — Score 3.0

Strong: Four-layer spec system. Per-group container.json (skills, provider, MCP servers, ERP gateway, resource caps, memory mode), composed CLAUDE.md (shared base + skill fragments + per-module instructions), runtime system-prompt addendum (live destination map, memory policy), and engage wiring in messaging_group_agents. Skills are first-class citizens (.agents/skills/<name>/SKILL.md) with YAML frontmatter and discoverable instruction fragments.

Gap: No version field on agent specs, prompts, or container.json. Edits overwrite in place. There is no agent-spec audit trail, no diff view, no PR-style approval workflow, no eval-on-merge gating. Compare to LangSmith Prompt Hub or Confident AI's git-based prompt management with branching and approval workflows.

To advance to 4.0: add agent_version and prompt_hash columns to agent_groups and messaging_group_agents; persist a snapshot of composed CLAUDE.md per session start; expose a diff view.

2. Short-term Memory (within-session) — Score 4.0

Strong: Dual-SQLite design (inbound.db host-owned, outbound.db container-owned) with even-odd seq numbering eliminates write contention. processing_ack table tracks message lifecycle, container_state exposes current tool-in-flight for stuck detection, exponential backoff (5s/10s/20s/40s, max 5 tries) on retry. The Claude provider's PreCompact hook archives full conversation transcripts to /workspace/agent/conversations/ before SDK auto-compaction at 165k tokens. splitBatchByTurn() enforces per-batch identity to prevent cross-user pollution. The current.md rolling-handoff pattern from taac2026 is a useful reference, but MUAP's runtime working memory is genuinely best-in-class for a multi-tenant runtime.

Gap: No explicit context-window budgeting visible to the operator (only the SDK's compaction threshold). No replay of historical message sequences into a fresh session.

3. Long-term Memory (cross-session) — Score 2.5

Strong: Per-group CLAUDE.local.md provides simple filesystem memory that survives container restarts. ERP Gateway contract defines erp_memory_get and erp_memory_upsert MCP tools with namespaced subjects (user.profile, conversation.summary, etc.) and HMAC-signed audit. memoryMode=erp automatically disables Claude's auto-memory and inserts a memory-policy addendum at runtime. Session archive (TTL-based tar + optional hard-delete) provides cold storage.

Gap: No local vector store, no embedding pipeline, no semantic recall. The ERP gateway is external — MUAP ships only the contract, not the implementation. Compared to mem0 (91.6 LoCoMo / 93.4 LongMemEval), Letta (core/archival/recall tiers), Zep Graphiti (temporal entity tracking), or Mastra Observational Memory (94.87% LongMemEval without vector DB), MUAP's long-term-memory story depends entirely on what the deployer builds behind the gateway. There is no 4-tier architecture (working / session / episodic / semantic).

To advance to 4.0: ship a reference ERP-memory-gateway implementation with at minimum keyword + temporal recall; add an embedding store at the gateway boundary; document a 3-tier policy (session-scoped working, user-scoped episodic, group-scoped semantic).

4. Evaluation Harness — Score 0.5 (Largest Gap)

Strong: A MockProvider exists (container/agent-runner/src/integration.test.ts). The classification_log table records every frontdesk routing decision with confidence, candidates, reasoning, and an outcome_ref linking to the actual delivered outbound action — this is explicitly seeded as "the regression-suite seed dataset" in source comments.

Gap: No actual eval harness consumes that seed. There are zero golden-dataset runs, zero LLM-as-judge calibration, zero rubric-based scoring, zero tau-bench / WebArena style trajectory evals, zero automated regression detection. Compared to Inspect AI (107 evals, 25 categories), DeepEval (50+ research-backed metrics + pytest CI gating), Promptfoo (red-team mode), or Braintrust (eval → release enforcement), MUAP has only unit tests of platform plumbing — not behavioral correctness evals.

To advance to 3.0: wire DeepEval or Promptfoo as a CI step against the classification_log corpus; add an evals/ directory with at least 50 golden cases per worker; gate Angel-style platform feedback into a reports/ lineage table similar to taac2026's pattern.

5. Observability (logs / metrics / tracing) — Score 3.0

Strong: Prometheus integration via prom-client exposes 10 custom metrics (inboundTotal, sessionCount, wakeRejectedTotal, routeSeconds histogram with 13 buckets, providerErrorsTotal, classificationsTotal, classificationBypassTotal, containerExitsTotal, sessionLifecycleTotal, classificationLogFailuresTotal) plus Node defaults at GET /metrics. Custom 5-level logger (src/log.ts) with LOG_LEVEL env, used in 45+ modules. Audit trails for ERP, enterprise events, classification, dropped messages, and inbound dedup.

Gap: Logging is colored text, not structured JSON — not log-aggregation ready (no ndjson, no Loki/Elastic shape). Zero distributed tracing — no OpenTelemetry, no trace IDs spanning host → container → ERP gateway → channel adapter. Compared to Mastra's OpenTelemetry-native bidirectional context propagation or LangSmith's hierarchical trace spans, MUAP cannot reconstruct a full request causal chain.

To advance to 4.0: swap the custom logger for pino with JSON output; instrument with OpenTelemetry; propagate trace_id end-to-end through the message lifecycle.

6. Multi-tenant Session Isolation — Score 4.5 (Best-in-class)

Strong: Five session modes (shared, per-thread, agent-shared, per-user, per-user-per-thread) implemented in resolveSession(). Container per session with Docker resource limits (memoryMb, cpus, pidsLimit) — far ahead of the namespace-filter pattern most platforms use. Per-session SQLite files (no shared DB writes between agents). RequestIdentity with splitBatchByTurn() derives trusted identity from origin_user_id (a2a) or senderId (channel) — prevents cross-user attribution in shared sessions. OneCLI gateway provides transparent credential injection without leaking secrets to agents. a2a hops propagate origin_user_id and root_session_id for end-to-end identity. unknown_sender_policy enforces sender-scope gates.

Gap: Containers share a kernel — for fully untrusted code execution, microVMs (Firecracker / Kata Containers) are the 2026 SOTA per Blaxel and Northflank. MUAP runs trusted code (only platform-defined skills), so this is acceptable but worth noting if untrusted-customer-code execution is ever in scope.

7. Audit & Replay — Score 3.0

Strong: Three durable audit tables (erp_audit, enterprise_audit, classification_log) plus operational tables (unregistered_senders, inbound_dedup, delivered). Every ERP gateway call is logged with input hash and identity boundary. Classification reconciliation stamps outcome_ref on the log row when the outbound action matches the declared classification — closes the audit loop.

Gap: No end-to-end correlation IDs (each table has its own id space, no spanning trace ID). No session replay primitive — /clear resets continuation, but cannot re-run a historical message sequence into a fresh session. Tool calls beyond ERP are not centrally logged.

8. Configuration & Schema Versioning — Score 3.0

Strong: 24 numbered DB migrations with schema_version tracking, name-keyed for module-safe ordering. container/Dockerfile pins CLAUDE_CODE_VERSION=2.1.116, BUN_VERSION=1.3.12, PNPM_VERSION=10.33.0, VERCEL_VERSION=52.2.1. Per-checkout install slug (frontlane-agent-v2-<sha1>:latest) prevents image collisions across multiple installs. Per-agent-group image building with imageTag override.

Gap: AGENT_BROWSER_VERSION=latest is an anti-pattern (reproducibility hazard). container.json edits via install_packages / add_mcp_server overwrite in place — no history. Per-session DBs use ad-hoc PRAGMA table_info() ALTERs rather than a migration runner.

9. Testing — Score 2.5

Strong: 53 test files, ~624 describe/it/test assertions. Vitest 4 (host) + Bun test (container). One real integration test exercising poll loop through MockProvider. Solid unit coverage of router, delivery, channels, DB, permissions, scheduling, sweeps, identity guard, MCP tools.

Gap: No coverage configuration (@vitest/coverage-* only present transitively in lockfile). Zero E2E tests covering channel → router → container → delivery → channel round-trip. Zero load/performance/stress tests. No __tests__/ or tests/ organization (tests co-located with source — fine, but no separation between unit and integration).

10. CI/CD & Health — Score 1.5

Strong: Lint (ESLint 9 + custom no-catch-all rule), typecheck (TypeScript strict), format (Prettier 3) all available as pnpm scripts. Crash-based circuit breaker (circuit-breaker.ts, file-persisted backoff up to 15 min, 1-hour reset). Graceful SIGTERM / SIGINT shutdown. Heartbeat-file-based stuck-container detection.

Gap: Zero CI infrastructure — no .github/workflows/, no .gitlab-ci.yml, no .circleci/, no pre-commit hooks. No /healthz, /ready, or /ping endpoint (/metrics could serve as liveness proxy but isn't designed for it). No Docker HEALTHCHECK.

Reference: taac2026 Patterns Worth Borrowing

The taac2026 ML competition project demonstrates a strong meta-discipline that MUAP could adopt without changing runtime semantics:

  1. Weak-Point Ledger (memory/weak_point_ledger.md) — every failure mode gets an entry with runs, root cause, pruning rule, and escape condition. Prunes future hypothesis branches before they consume compute. MUAP equivalent: a failure_modes.md listing classifier-bypass causes, container-stuck patterns, ERP-gateway-failure shapes — each with a stop rule.

  2. Idea Board (scratchpad/idea_board.md) — 2-D feature-family × iteration grid with breadth/depth budget enforcement. MUAP equivalent: per-release backlog showing which subsystems are receiving change pressure simultaneously, capping concurrent areas of churn.

  3. SMOKE Contract — every submission package ships with SMOKE.py doing byte-identity checks against the parent, parameter-count assertions, NaN/gradient checks. MUAP equivalent: a per-release smoke pack that re-verifies the dual-SQLite invariants, batch-identity guard, classification reconciliation loop.

  4. Course Library Schema — memory/courses/<topic>/{README, code_blocks, case_studies}.md with Source / Date / Rationale / Status metadata and explicit expansion-gating thresholds. MUAP equivalent: a docs/patterns/ registry where each architectural pattern (engage modes, session isolation, a2a routing, ERP memory) has a versioned spec page.

  5. Three-Layer Memory — AGENTS.md (rules) → memory/*.md (durable knowledge) → scratchpad/current.md (rolling handoff). MUAP already has CLAUDE.md (rules) and CLAUDE.local.md (durable). Missing: a host-side rolling-handoff equivalent for operators (changelog, in-flight migration status).

  6. TensorBoard Scalar Contract — taac2026 mandates 15 specific scalar tags every trainer must emit, because the platform doesn't preserve logs. MUAP equivalent: mandate a fixed Prometheus metric name set per agent group, so dashboards survive worker turnover.

  7. Iteration Safety Ladder — Tier 1 (training dynamics) → Tier 4 (structural changes), each with gating. MUAP equivalent: tier change-control on container.json edits — Tier 1 (resource caps) auto-approved, Tier 4 (skill set, MCP servers, memory mode) requires admin approval.
